# Output Patterns

Use these formats when the user needs a concrete deliverable. Adapt headings to the request instead of forcing every section.

## Product Critique

```text
Top judgment:
Assumptions:
Current user goal:
Key friction points:
High-leverage recommendations:
Measurement plan:
Risks and open questions:
```

For each recommendation, include:

- What to change.
- Why it matters.
- Expected user impact.
- Expected metric impact.
- Tradeoff or risk.

## Design Brief

```text
Problem:
Audience:
User job:
Product goal:
Non-goals:
Constraints:
Core flow:
Experience principles:
Key screens or states:
Success metrics:
Validation plan:
Risks:
```

## PRD Or Product Plan

```text
Opportunity:
Evidence:
Goal:
Users and use cases:
MVP scope:
Out of scope:
Functional requirements:
UX states:
Instrumentation:
Launch or rollout:
Risks:
Open decisions:
```

Keep PRDs sharp. Remove sections that do not change decisions.

## Experiment Or Validation Plan

```text
Hypothesis:
Audience:
Prototype or product change:
Primary metric:
Guardrails:
Method:
Sample or duration:
Decision rule:
Follow-up if it wins:
Follow-up if it loses:
```

## Portfolio Or Case Study Narrative

Use this when helping with product design portfolios, interview stories, or case studies:

```text
One-line project thesis:
Context and role:
Problem evidence:
Key constraint:
Design strategy:
Important decisions:
Data or research used:
Outcome:
Learning:
```

Make the designer's judgment visible. Avoid vague claims like "improved UX" without naming the product decision, evidence, and impact.

## Interview-Style Product Reasoning

When answering a product design interview prompt:

1. Clarify the product goal and user segment briefly.
2. State assumptions.
3. Identify user needs and business constraints.
4. Propose a focused solution direction.
5. Explain tradeoffs.
6. Define success metrics and validation.
7. Mention edge cases or risks.

Do not overcomplicate the answer with full frameworks unless the prompt asks for depth.
